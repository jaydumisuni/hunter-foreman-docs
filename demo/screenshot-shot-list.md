# Demo Screenshot Shot List

These are the final screenshots to capture for the hackathon submission page.

## Required screenshots

### 1. Hero / landing screen

Purpose:

- Shows Hunter Foreman identity.
- Establishes that this is not only a chatbot.
- Should include the strongest visible product message.

Capture:

```text
Main app home / landing page
```

### 2. ROSE intake

Purpose:

- Shows the customer request entry point.
- Demonstrates that requests start from a business-facing intake flow.

Capture:

```text
ROSE form or chat/intake panel with a realistic customer request typed in
```

Suggested request:

```text
We need wedding invitations, tickets, and QR check-in support for a conference.
```

### 3. Task created / classification result

Purpose:

- Shows that Foreman turns a request into structured work.
- Should show intent, confidence, escalation state, and fallback/provider metadata if visible.

Capture:

```text
Task detail or dashboard card after submitting the ROSE request
```

### 4. Dashboard state

Purpose:

- Shows operational visibility.
- Judges should see that the work has a state, not only a reply.

Capture:

```text
Dashboard with created task, state ladder, timeline, or active work view
```

### 5. App bridge / connected dispatch

Purpose:

- Shows the external app bridge contract in action.
- Demonstrates that Foreman can move work to another application.

Capture:

```text
Bridge status or dispatch confirmation in the main app
```

### 6. Receiver acknowledgement

Purpose:

- Shows connected receiver app accepting the task.
- This is the key proof that the workflow crosses app boundaries.

Capture:

```text
hunter-foreman-demo receiver page showing received task and acknowledgement
```

### 7. Proof / repository evidence

Purpose:

- Shows seriousness and reproducibility.
- Useful if the platform allows extra images.

Capture one of:

```text
GitHub Actions Submission Proof success
Clean repo list / README fast review path
Sergeant proof layer README or merged PR status
```

## Optional screenshots

- Fireworks live attempt report showing honest claim boundary.
- Public safety boundary document.
- System overview diagram.
- Final freeze gate.

## Screenshot rules

- No API keys.
- No private customer data.
- No private Hunter internals.
- No payment secrets.
- No local machine secrets in terminal.
- Prefer browser UI over terminal output for judge-facing screenshots.

## Minimum set

If time is short, capture only these four:

1. ROSE intake
2. Task created / dashboard
3. Receiver acknowledgement
4. Submission Proof success
